# Короткий текст защиты

Я участвовал в технической проработке проекта команды «Прайм» по программному управлению роботами.

Мой вклад был связан не только с финальным запуском машинки, а именно с процессом разработки: анализом модели объекта, переходом к матричной форме, проверкой закона управления, отладкой связи между сайтом и Arduino, а также обработкой проблем, которые возникали на железе.

В проекте были последовательно решены несколько задач. Сначала регулятор следования был вынесен из `server.py` в отдельные файлы `controllers/follow_base.py`, `controllers/follow_car1.py` и `controllers/follow_car2.py`. Это позволило сделать отдельный алгоритм для каждой машинки и запускать два регулятора параллельно.

Потом была исправлена проблема с PWM. В нашей прошивке логика инверсная: `PWM = 0` соответствует максимальной скорости, а `PWM = 255` — остановке. Поэтому серверная часть была приведена к этой логике, чтобы команды скорости не уходили в неправильный режим.

Дальше закон управления был переписан в матричном виде. В коде управление считается не набором отдельных условий, а через выражения вида `q = F*xi`, `u = S*q`. Служебные `if` остались только для обработки ошибок, подключения и логирования, но не внутри математического закона управления.

Также был доработан MPU6050: чтение гироскопа переведено на библиотеку I2Cdev, калибровка выполняется один раз при первом обращении, а функция возвращает угловую скорость в rad/s. Это нужно для более корректной оценки угла рыскания.

В итоговой версии модель объекта записана как дискретная матричная модель:

```text
x(k+1) = Ad*x(k) + Bd*u(k) + Gd*vf(k) + Hd*omega_gyro(k)
y(k)   = C*x(k)
```

А закон управления:

```text
q(k) = F*xi(k)
u(k) = S*q(k)
PWM(k) = 255*(1-u(k))
```

В репозитории сохранены все версии проекта, changelog, описание правок и исходные архивы. Это показывает не только результат, но и реальный процесс работы: какие ошибки находились, как они исправлялись и как код постепенно приводился к математической модели.
