# Выполненные доработки проекта

## 1. Разделение серверной логики и регулятора

В ранней структуре часть логики следования была связана с сервером. В обновлённой структуре сервер занимается подключениями и обменом, а регуляторы вынесены в отдельные файлы.

Результат:

- код стал проще разделять по назначению;
- регулятор можно менять без переписывания маршрутов Flask;
- для двух машинок можно использовать разные параметры.

## 2. Поддержка двух машинок

В проект добавлена структура для работы с двумя машинками:

```text
car1
car2
```

Каждая машинка имеет свой регулятор. Это позволяет запускать управление параллельно, а не выполнять алгоритмы по очереди.

## 3. Поддержка одной машинки

Отдельно исправлена ситуация, когда система ожидала обязательного подключения второй машинки. Теперь возможны режимы:

```text
только car1
только car2
car1 и car2
```

Это полезно для отладки: можно проверять одну машинку без полного стенда.

## 4. Исправление PWM

В используемой прошивке логика PWM инверсная:

```text
0   -> максимальная скорость
255 -> стоп
```

После исправления перевод управления в PWM соответствует реальному поведению моторов. Это устраняет ситуацию, когда команда скорости могла интерпретироваться наоборот.

## 5. Обработка временных ошибок дальномера

Дальномер может кратковременно потерять сигнал. В регуляторе это учитывается как возможная временная ошибка измерения, а не как причина полностью останавливать алгоритм.

Результат:

- меньше аварийных остановок;
- регулятор устойчивее к одиночным сбоям датчика;
- поведение системы становится более предсказуемым.

## 6. MPU6050 через I2Cdev

Для гироскопа подготовлена версия с библиотекой I2Cdev. Калибровка выполняется один раз при первом обращении к датчику.

Результат:

- калибровка не повторяется в каждом цикле;
- уменьшается задержка измерений;
- сервер получает угловую скорость в нормальных единицах.

## 7. Матричная форма закона управления

Закон управления переписан через матричные операции:

```text
q = F*xi
u = S*q
```

Служебные условия остаются в инфраструктуре программы, но формирование управляющего воздействия выполнено через матрицы.

## 8. Матричная модель объекта

Для итоговой версии добавлена запись состояния:

```text
x(k+1) = Ad*x(k) + Bd*u(k) + Gd*vf(k)
y(k)   = C*x(k)
```

Это связывает программную реализацию с моделью объекта и упрощает дальнейшую настройку регулятора.

## 9. Объединённый опрос датчиков

Команда `sensors` объединяет получение расстояния и угловой скорости. Это уменьшает число отдельных запросов в цикле управления.

## 10. Журнал версий

В репозитории сохранены промежуточные версии проекта. Это удобно для сравнения изменений и возврата к предыдущим решениям при отладке.
