# Досье участника проекта

**Участник:** SouthBoard  
**Команда:** «Прайм»  
**Проект:** программное управление мобильными роботами  
**Направление работы:** сервер управления, регуляторы следования, матричная модель, Arduino/ESP, датчики

## 1. Участок работы

В проекте я занимался технической частью, связанной с переходом от отдельных рабочих версий к более цельной структуре проекта. Основное внимание было на серверной логике, регуляторах следования, связи с Arduino/ESP и оформлении математической модели управления.

Моя задача состояла в том, чтобы связать между собой несколько частей проекта:

- веб-интерфейс управления;
- Flask-сервер;
- подключение одной или двух машинок;
- получение данных с датчиков;
- формирование управляющего воздействия;
- перевод управления в PWM;
- описание модели объекта.

## 2. Работа с архитектурой проекта

В ранних версиях часть логики регулятора была связана с кодом сервера. В дальнейшем регуляторы были вынесены в отдельные файлы:

```text
controllers/follow_base.py
controllers/follow_car1.py
controllers/follow_car2.py
```

Такой вариант удобнее, потому что сервер отвечает за соединения, маршруты и обмен данными, а регуляторы — за расчёт управляющих воздействий.

Для двух машинок используется раздельная логика:

```text
car1 -> follow_car1.py
car2 -> follow_car2.py
```

Это позволяет изменять параметры регулятора для каждой машинки отдельно.

## 3. Работа с PWM

В ходе работы была обнаружена важная особенность прошивки: PWM используется в инверсной форме.

```text
PWM = 0   -> максимальная скорость
PWM = 255 -> остановка
```

Из-за этого стандартная формула Arduino не подходила. Логика перевода управления была исправлена так, чтобы нормированное управление `u` переводилось в PWM по формуле:

```text
PWM = 255 * (1 - u)
```

Это изменение важно, потому что неправильная интерпретация PWM приводила к некорректной реакции моторов на команды скорости.

## 4. Работа с датчиками

В проекте используются:

- HC-SR04 для измерения расстояния;
- MPU6050 для получения угловой скорости;
- ESP/Arduino для связи с сервером и выполнения команд.

Для MPU6050 была подготовлена версия с библиотекой I2Cdev. Калибровка гироскопа выполняется один раз при запуске, чтобы не тратить время на калибровку при каждом измерении.

Также была добавлена идея объединённой команды `sensors`, чтобы за один запрос получать основные данные датчиков.

## 5. Матричная модель и закон управления

Для описания объекта использовалась матричная форма:

```text
x(k+1) = Ad*x(k) + Bd*u(k) + Gd*vf(k)
y(k)   = C*x(k)
```

Закон управления был приведён к виду:

```text
q = F*xi
u = S*q
```

Такой подход отделяет математический расчёт управления от служебной логики программы. Ветвления остаются в инфраструктуре: запуск, остановка, обработка ошибок, чтение датчиков. Само управляющее воздействие формируется через матрицы.

## 6. Работа с одной и двумя машинками

Была исправлена ситуация, когда система ожидала обязательного подключения двух машинок. В обновлённой логике возможны три режима:

```text
только car1
только car2
car1 и car2 одновременно
```

Если подключена одна машинка, сервер запускает регулятор только для неё. Если подключены две машинки, регуляторы могут запускаться параллельно.

## 7. Результат

В результате была собрана структура проекта, где есть:

- финальная версия кода;
- сохранённые промежуточные версии;
- журнал изменений;
- описание назначения файлов;
- технические заметки по памяти Arduino;
- описание матричной модели;
- пояснение ключевых частей кода.

Такой формат удобен для дальнейшей доработки проекта: можно понять, какие изменения были сделаны, какие проблемы решались и где находится каждая важная часть системы.
